D — 페이지 번호 함께 저장

UI 변화 0. sessionStorage에 {y, page} 저장. 복원 시 page까지 미리 fetch한 후 scrollTo

1️⃣ 사용자가 50번째 카드 클릭 직전 sessionStorage 저장: {y: 1500, page: 3}

2️⃣ 게시글 페이지 풀 전환 (UI는 Before와 동일)

3️⃣ 뒤로가기 → 메인 마운트 시 sessionStorage 읽음 → page 3까지 sequential하게 fetch (3번 API 호출)

4️⃣ 콘텐츠 다 로드된 후 scrollTo(0, 1500) → 정확한 위치 ✓

⚠️ 단점: 페이지 3번 fetch라 약간 느림 (300~600ms 로딩). UI는 그대로지만 "복귀가 느려진다"는 단점

UI 변화: 0. 차이는 fetch 횟수: 무한 스크롤 page만큼 다시 fetch. 작업량 3~4h. 라이브러리 추가 X. 다만 fetch가 늘어나 백엔드 부담

📌 결정 가이드

A

최대한 우아한 UX. 게시글이 모달로 떠도 괜찮고, 6~8h 작업이 OK라면 → A안. Reddit/Instagram의 "그 부드러움" 그대로 재현

B

UI는 그대로 두고 싶고, TanStack Query 같은 캐싱 라이브러리를 도입할 의향 있다면 → B안. 다른 영역에도 캐싱 효과 + 4~5h 작업

D

UI 그대로 + 라이브러리 추가도 싫다면 → D안. 다만 복귀가 살짝 느려짐(여러 페이지 sequential fetch)

💡 제 우선순위 추천: A(가장 우아) > B(데이터 캐싱) > D(작은 변경) > C(현재 미세 개선, 점프 보일 수 있음)

UI 변화 허용 여부가 가장 큰 결정 기준입니다. A안의 모달 UI가 마음에 들지 않으면 B/D, A안을 받아들일 수 있다면 가장 부드러운 UX가 나옴.
